# UI_GUIDELINES.md

## 1. 文档目标

统一本项目前端与小程序的视觉与交互标准，避免“每个页面像不同产品”。  
适用范围：`jiajiao-admin-web`、`mp-weixin-jiajiao`。

## 2. 设计原则

- 一致性：相同语义使用相同视觉表达。
- 清晰性：让用户一眼知道“当前状态 + 下一步动作”。
- 克制性：不堆砌装饰，不滥用颜色和动效。
- 可维护性：优先复用组件与样式令牌。

## 3. 视觉基础规范

### 3.1 色彩

- 语义色建议固定为：
  - 主色（Primary）
  - 成功（Success）
  - 警告（Warning）
  - 错误（Danger）
  - 信息（Info）
- 同一语义色在全站保持统一，不要页面级“重新发明”。
- 禁止使用“仅靠颜色表达状态”，需辅以图标/文案。

### 3.2 字体与层级

- 至少定义 4 级文本：
  - 页面标题
  - 区块标题
  - 正文
  - 辅助说明
- 优先保证可读性，避免过小字号与过低对比度。

### 3.3 间距与圆角

- 间距采用统一节奏（推荐 4/8 体系）。
- 圆角按组件类型固定（如按钮、卡片、输入框）。
- 禁止随意写“奇怪像素值”（例如 13px、19px 这类无体系值）。

### 3.4 阴影与边框

- 阴影仅用于层级区分，保持轻量。
- 优先用边框做结构分割，避免过重阴影导致“悬浮感过强”。

## 4. 组件规范

### 4.1 按钮

- 必须区分主按钮与次按钮。
- 状态完整：默认、悬停、激活、禁用、加载。
- 同一区域避免出现多个“主按钮”抢焦点。

### 4.2 表单

- 标签、占位、错误提示风格统一。
- 必填项要有明确标识。
- 校验失败文案要告诉用户“怎么改”。

### 4.3 表格与列表

- 列名语义清晰，避免内部字段名直接暴露。
- 空态必须有引导（不是只显示空白）。
- 操作列优先级明确，危险操作需二次确认。

### 4.4 弹窗与抽屉

- 只用于需要聚焦的任务。
- 标题应直接说明动作目标（如“审核教师资料”）。
- 关闭路径明确（取消、关闭按钮、Esc）。

## 5. 页面状态规范

每个页面至少覆盖：

- `loading`：首次加载骨架或加载提示
- `empty`：无数据时的友好引导
- `error`：请求失败时可重试
- `success`：操作成功有明确反馈

## 6. 交互与动效规范

- 动效必须“有意义”，例如：
  - 页面切换过渡
  - 列表项渐进出现
  - 按钮提交 loading
- 禁止炫技型复杂动画，避免影响效率与可读性。
- 动效时长建议短且稳定，避免忽快忽慢。

## 7. 响应式与多端适配

- 管理后台至少保证：
  - 常见桌面分辨率正常
  - 窄屏下布局不崩溃（可降级）
- 小程序重点保证：
  - 主流程在常见机型可操作
  - 点击区域足够大，避免误触

## 8. 可访问性基线

- 文本与背景保持足够对比度。
- 关键按钮可通过文案独立理解，不依赖颜色。
- 点击元素可触达面积合理（移动端尤其重要）。

## 9. 前端实现建议

- 管理后台：
  - 建议在 `src/style.css` 或独立 token 文件统一维护颜色/间距变量
  - 新组件优先放 `src/components`，避免页面内重复实现
- 小程序：
  - 通用样式优先放公共样式层
  - 公共交互逻辑优先放 `utils` 或公共组件

## 10. 禁止事项（高频问题）

- 禁止复制粘贴式重复 UI 代码，不抽公共组件。
- 禁止页面间同语义不同样式（按钮、标签、状态色混乱）。
- 禁止只做“视觉完成”，忽略加载/错误/空态。
- 禁止未经确认引入大型 UI 库导致包体膨胀。

## 11. 设计评审自检清单

提交前自检：

1. 信息层级是否一眼可读
2. 交互状态是否完整（loading/empty/error）
3. 是否复用已有组件与样式 token
4. 是否兼顾桌面与移动场景
5. 是否存在明显可访问性问题

## 12. 与 AGENTS.md 配合方式

新对话开始时，建议明确要求代理先阅读：

- `AGENTS.md`
- `PROJECT_CONTEXT.md`
- `API_CONTRACTS.md`
- `UI_GUIDELINES.md`

推荐话术：

```text

```
