# Frontend Fullstack UI

## 概述

以“可维护 + 可验证 + 可交付”为核心，完成从需求澄清到实现、测试和交付说明的完整前端流程。  
优先做最小可行改动，保持现有设计系统与代码风格一致。

## 快速执行流程

1. 澄清目标和约束  
2. 扫描相关代码与样式结构  
3. 输出实现方案（组件、状态、数据流、兼容策略）  
4. 实施改动并自测  
5. 输出变更清单、验证步骤和风险说明

## 工作流决策树

- 如果是“新页面/新模块”：优先使用组件化和样式令牌设计。
- 如果是“现有页面问题”：先定位根因（样式冲突、状态管理、接口契约、渲染性能）。
- 如果是“卡顿/慢”：先采样性能瓶颈，再做针对性优化。
- 如果是“兼容性”：先确认目标环境，再做最小兼容补丁。
- 如果是“小程序问题”：优先检查页面注册、路由参数、授权态与 API 基础地址。

## 任务策略

### 1) UI 与交互开发

- 先确定信息层级：标题、正文、操作区、反馈区。
- 统一间距、字号、颜色和状态表达，避免页面风格漂移。
- 组件状态要完整：默认、悬停、激活、禁用、加载、空态、错误态。
- 保持语义化结构与可访问性。

### 2) Vue + Vite 开发与优化

- 优先复用已有组件与 API 封装。
- 路由变更时同步检查导航守卫和菜单可见性。
- 避免在组件中堆积业务逻辑；提取到 composable/工具层（若项目已有该模式）。
- 关注打包体积、重复渲染、无效依赖和大对象响应式开销。

### 3) 微信小程序开发与优化

- 检查 `app.json` 页面注册、分包和 tabBar 配置。
- 检查授权态与 token 失效处理。
- 检查请求基础地址、超时、重试、错误提示文案。
- 关注首屏渲染、图片体积、setData 粒度和频率。

### 4) 浏览器兼容与稳定性

- 优先确认目标浏览器范围，再补兼容。
- 优先修复真实用户影响路径，不盲目引入重型 polyfill。
- 对 CSS 特性和布局差异提供降级方案。
- 对事件、输入法、滚动、视口相关问题做环境测试。

### 5) 性能优化

- 先测量后优化：首屏、交互响应、网络请求、长任务。
- 优先处理高收益项：
  - 减少不必要重渲染
  - 拆分大组件与大资源
  - 图片压缩与懒加载
  - 缓存与请求合并
- 每次优化要给出“优化前后对比或可观察变化”。

## 实施要求

- 先阅读项目根目录的 `AGENTS.md`（若存在）。
- 只修改任务相关文件，不做无关重构。
- 保持接口契约一致；字段变更时同步调用端。
- 无法完成的项要明确阻塞原因与建议下一步。

## 验证清单

执行任务后至少完成以下检查：

- 构建可通过（例如 `npm run build`）。
- 关键页面可打开，核心交互可走通。
- API 成功与失败路径都有反馈。
- 不引入明显控制台报错。
- 移动端/小程序关键页面无严重布局错位。

读取更详细检查项：`references/frontend-checklists.md`

## 输出模板

按以下结构汇报结果：

1. 改动结论（1-3 句）
2. 修改文件清单
3. 验证步骤（命令 + 页面路径）
4. 风险与后续建议（若有）
